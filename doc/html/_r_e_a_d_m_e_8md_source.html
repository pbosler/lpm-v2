21 Design / Use
22 =========
23 
24 The code is organized in an object-oriented style using modern Fortran.
25 Hence, there is a class (Fortran derived data type) that corresponds to every task that the code completes.
26 Generally, each data type has its own module file.@n
27 A C++ implementation is in development.

29 Base objects
30 ------------
31 LPM provides a small library of fundamental modules that are used by the rest of the code.
32 
33 * The @ref NumberKinds module defines constants used by the rest of the code, including variable kind definitions and physical constants.
34 * The @ref OutputWriter class defines methods for formatting output either for the console or ASCII files.
35 * The @ref Logger class provides objects and methods for outputting to the console various types of messages from the rest of the LPM code in a parallel computing environment.
36 * The @ref SphereGeom and @ref PlaneGeom modules define geometric formulas for distance and areas.
37 * The @ref MPISetup module provides a type and methods for distribution other objects across MPI ranks.
38 * The @ref STDIntVector module provides a data type that mimics the C++ `std::vector<int>` class to dynamically allocate and resize integer arrays of rank 1

40 Particles and Fields
41 --------------------
42 The primary computing objects used by LPM are the Lagrangian particles, defined in the particles module, and the fields, defined in the field module.
43 * @ref Particles provide the spatial discretization of a domain and have both physical and Lagrangian coordinates.
44 * A @ref Field may be either a scalar or vector field defined on a set of particles; fields are used to store variables (e.g., vorticity, temperature, etc.) on a set of particles.

47 Mesh objects
48 ------------
49 While the dynamics and PDEs are solved by LPM in a ''meshfree'' context, we often find it helpful and efficient to maintain a mesh to organize our particles.
50 Topologically two-dimensional meshes (e.g., spheres and planes) are currently implemented using triangles and quadrilaterals.
51 * The @ref Edges module defines directed edges that connect an origin particle to a destination particle, and have a left face and a right face.
52 * The @ref Faces module defines polyhedral faces that have an associated list of edges and vertices.
53 * The @ref PolyMesh2d module collects particles, edges, and faces, into a single mesh object.
54 Meshes are initialized from a ''seed'' that is recursively divided until a desired spatial resolution is achieved.
55 
56 * @ref PolyMesh2d meshes may be adaptively refined using the @ref Refinement module.
57 * Remesh/remap subroutines depend on a separate scattered data algorithm. Current implementations include @ref BIVARRemesh for planar problems
58 and @ref SSRFPACKRemesh for spherical problems.
59 
60 External libraries
61 -------------------
62 LPM code requires scattered data interpolation or approximation algorithms to perform its remesh/remap step.
63 For planar applications, the bivariate quintic Hermite polynomials of the BIVAR pacakge (H. Akima, _ACM TOMS_, 1978) are used (bivar.f90).
64 For spherical applications, the cubic Hermite polynomials with exponential tension factors of the STRIPACK/SSRFPACK libraries (Renka, _ACM TOMS_, 1997) are used (stripack.f and ssrfpack.f).
65 
66 Other methods for scattered data approximation are under development.
67 
68 PDE types and Solver objects
69 --------------
70 PDEs are solved using a ''method of lines'' discretization.
71 A mesh is connected to its relevant variables by a PDE type, e.g., an SWEMesh or a @ref SphereBVE BVEMesh object.
72 These objects combine a PolyMesh2d for the spatial discretization with the various Fields (e.g., vorticity, temperature)
73 appropriate to the chosen PDE.
74 
75 Following the spatial discretization given by the particles, the time-ODEs are integrated explicity with 4th order Runge-Kutta by a Solver
76 (e.g., @ref SphereBVESolver) object appropriate to a particular application, e.g., SWEPlaneSolver or SphereBVESolver.
77 
78 * Planar, inviscid, incompressible flow: See @ref PlanarIncompressible, @ref PlanarIncompressibleSolver, and CollidingDipoles.f90
79 * Rotating planar, inviscid, incompressible flow: See @ref BetaPlane, @ref BetaPlaneSolver
80 * Spherical Barotropic Vorticity Equation (BVE): See @ref SphereBVE, @ref SphereBVESolver, and BVESingleGaussianVortex.f90
81 * Rotating planar Shallow Water Equations (SWE): See @ref PlanarSWE, @ref SWEPlaneSolver, NitscheStrickland.f90, etc.

83 Application Drivers
84 --------
85 Finally, all of the above objects are assembled by a driver program that solves a specific problem.
86 Within the driver program all initial conditions must be defined.
87 
88 Run-time user input to driver programs is handled via namelist file, that is typically supplied as the first argument on the command line.
89 See below for more discussion.
90 
91 It is within these driver programs that the timestepping loop and remeshing procedures are called.
92 See PlaneSWEGravityWaves.f90 or CollidingDipoles.f90 for examples.

95 Build / Install
96 ================
97 This latest version of LPM is written with support for KitWare's [CMake](http://www.cmake.org) cross-platform makefile generator, to aid
98 portability and ease installation for new users.
99 
100 LPM requires an [OpenMPI](http://www.open-mpi.org) distribution for parallel computing.
101 
102 The environmental variables `CC`, `CXX`, and `FC` specify the compilers.
103 For example, using the bash shell, these variables are set with
104 
105  export CC=mpicc
106  export CXX=mpicxx
107  export FC=mpifort
108 
109 LPM is configured in the same shell by navigating to the lpm-v2 root directory and typing
110  mkdir build
111  cd build
112  cmake ..
113 This configures the build for the your compiler version and MPI. Build the LPM software next by typing
114 
115  make
116 
117 then
118 
119  make test
120 
121 to run the unit tests.

123 Running applications
124 ====================
125 
126 LPM uses Fortran's handy namelist utility to specify run-time input to programs.
127 
128 To run an application driver, LPM executables may be run with the command
129 
130  mpirun -np 6 <LPMfile>.exe <namelist file>
131 
132 This would execute the specified .exe file using 6 MPI processes and the input variables defined in the specified namelist file.

134 Driver programs
135 ----------------
136 
137 Each application is defined in its own driver program.
138 Examples include `CollidingDipoles.f90`, `BVESingleGaussianVortex.f90`, and `PlaneSWEGravityWaves.f90`. @n
139 
140 Driver programs are organized into three phases:
141 1. __Initialize:__ The computing environment is defined and initialized using appropriate calls to MPI subroutines. @n
142  User input is read from the appropriate namelist file. @n
143  The initial set of particles and their initial mesh are defined. @n
144  Initial conditions for each variable are defined on the mesh. @n
145  Any initial adaptive refinement is completed. @n
146  The initial conditions are output to data files. @n
147 2. __Run:__ The time step loop resides in the section of the driver. @n
148  Within the timestep loop, an `if` statement determinces whether or not a particle set needs to be remeshed and/or remapped. @n
149  Remeshing (if necessary) is completed. @n
150  Time is advanced by one time step for each iteration of the loop. @n
151  Time-dependent output is written to a data file. @n
152 3. __Finalize:__ Final output is written to data files. @n
153  Performance and timing data are output to console. @n
154  All LPM objects are deleted and all memory used by the driver is freed.
155 
156 Within the `contains` section of a driver program, users may insert functions relevant to their application. @n
157 In most cases, these functions must conform to the interfaces defined in @ref NumberKinds and @ref Refinement . @n
158 Additionally, since different applications will require different input, each driver requires its own version of the `ReadNamelistFile` subroutine.
